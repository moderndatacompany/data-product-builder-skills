# About

## **Why "*Vulcan*"**

We're a semantics-core product, so the name isn't incidental. ***Vulcan*** is the Roman god of the forge. Not of fire in the abstract, but of **fire put to work**. The forge is where raw ore becomes something precise and load-bearing.

That's the same transformation the Data Product Stack or *Vulcan* performs: raw SQL and Python go in, certified data products come out. The forge doesn't just heat things, but shapes them under controlled conditions, removes impurities, and produces something you can depend on structurally.

Scholars trace *Vulcanus* back to its roots, meaning lustre, brilliance, or glare.

> *Vulcan* is named for that quality: not the fire itself, but what fire reveals.

Visibility into what's running, what's queued, what failed, and why. A blacksmith doesn't guess whether the metal is ready because he can see it. Vulcan gives data teams the same ability: to see exactly what's happening at every step, before bad data reaches anyone who acts on it.

---

## What is *Vulcan*

*Vulcan* is a data product stack built for developers with a mission to enable data for direct business impact. The stack covers the **full journey from raw code to business-ready data products.** It handles CI/CD, testing, data quality enforcement, and API generation in **one coherent system**.

---

## **The *Vulcan* Mental Model**

Think of *Vulcan* as a contract system. Each model in your project makes implicit promises: that it will produce a certain shape of data, that its dependencies are satisfied, and that the output meets defined quality thresholds.

*Vulcan* makes those promises explicit and verifiable. When a model passes all its checks, it **becomes a "certified" data product**. That certification is what allows downstream consumers, whether APIs, semantic models, or business users, to trust what they receive **without inspecting how it was produced**.

This means you can reason about any part of the system in isolation.

> A semantic model doesn't need to know how its source models are implemented, only that they've been validated. An API doesn't need to know what the semantic layer ingested, only that it passed.

This is the same principle that makes interfaces useful in software engineering: you hide the implementation and expose a verified contract.
